核心痛点
订货偏差 = 真实损失
订货量 > 需求 → 报损（成本浪费）
订货量 < 需求 → 缺货（丢失销售机会、客户流失）
传统人工无法应对 数十万SKU × 千家门店 的组合爆炸。
核心目标
在降低报损的同时，将缺货率稳定控制在5%以内，实现动态精准补货。
架构亮点
  • ✅ 长链推理 — 6步推理链路，多阶段决策
  • ✅ 多Agent协作 — 并行预测+并行安全库存+节奏整合+异常仲裁
  • ✅ 防幻觉机制 — LLM仅用于自然语言解释，不参与数值订货
  • ✅ 规则仲裁模块 — 保障库存周转不低于红线
长链推理 (6步完整推理链路)
①
数据清洗模块
清洗历史销售、退货、剔除促销异常点 → 干净时序数据
②
预测 Agent
基于 LightGBM/Prophet 输出 T+14 日基准需求量（每个门店-SKU）
③
安全库存 Agent
需求标准差 σ × 服务水平系数(z=1.96) × √提前期 → 每日安全库存量
④
订货节奏 Agent
聚合预测+安全库存，减去当前库存和在途，按配送日历/最小包装倍数计算初步订货量
⑤
规则仲裁模块
当预测+安全导致周转天数<7天时，强制执行保底系数，避免过度压货
⑥
异常检测 Agent
校验最终订货量是否超过历史3σ阈值或小于0，拦截明显异常值
长链推理特征：全部步骤超过5个推理阶段，且每一步的输出都作为下一步的条件输入，形成深度决策流。
多 Agent 协作机制
Agent职责运行方式输出格式
预测 Agent时序模型预测T+14需求量并行(无状态){store_id, sku, date, pred_qty}
安全库存 Agent基于波动 & 服务水平计算安全库存并行(依赖统计){store_id, sku, date, safety_stock}
订货节奏 Agent聚合 + 库存抵消 + 配送规则串行等待前两者{order_qty, order_date}
异常检测 Agent最终校验与告警串行最后一步{approved: bool, reason}
冲突解决：当“预测+安全库存”组合导致库存周转天数下降过快（<7天），规则仲裁模块（非LLM）强制执行保底系数0.8，优先保障服务水平。
技术实现要点

Python 3.10 + Celery + Redis 分布式任务分片

时序模型：LightGBM / Prophet (按SKU类别自动选择)

安全库存公式：动态 z-score (服务水平 95%)

自研 DAG 引擎（并行 + 串行依赖编排）

LLM 仅用于异常告警解释 + 自然语言建议（不参与订货数值，防幻觉）

每日凌晨2点全量计算，覆盖210万+订单行

Docker Redis PostgreSQL 时序数据库
